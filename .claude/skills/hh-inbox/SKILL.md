# hh-inbox: обработка входящих по откликам

Рабочий каталог: `/home/max/automations/hh`.

## step-0: Режим

`interactive` (дефолт) — согласования через AskUserQuestion. `unattended` (`/hh-inbox auto`, либо юзер явно велел работать списком/не спрашивать) — всё из рунга СОГЛАСОВАНИЕ не отправлять: `status: awaiting_user` + позиция в секцию «Ждёт юзера»; рунги МОЛЧАНИЕ и АВТО — как обычно.

## step-1: Сбор

```bash
python3 src/inbox.py
```

→ `data/inbox.json` (прошлый снимок ротируется в `inbox-prev.json`). Покрытие полное: chatik-конфиг даёт последние ~20 чатов с текстом lastMessage, остальное добирает negotiations-пагинация (`source: negotiations`, без текста — его дочитает субагент step-3 из чата). Текст lastMessage обрезан до 2000 симв.

Обрабатывать: `needsAttention: true` (новое чужое сообщение после `lastContact`; unread не годится — гаснет от чтения с телефона) + `stateChanged` из сводки.

## step-2: Классификация (главный тред; сверху вниз, first match)

1. **МОЛЧАНИЕ** — только лог, в чат не писать и не кликать НИЧЕГО:
   - диалог закрыт ботом: «ответы отправлены работодателю», «ИИ-помощник завершил работу», итоговая сводка ответов, «пользователь … покинул чат» — даже если выше висят неотвеченные вопросы (потребовано юзером@2026-07-19: ответ никуда не попадёт);
   - отказ: `state: DISCARD`, «не готовы пригласить» → `status: rejected`; никаких поправок/уточнений задним числом (потребовано юзером@2026-07-13);
   - инфо без вопроса («передали рекрутеру», «рассмотрим»).
2. **СОГЛАСОВАНИЕ** (interactive: AskUserQuestion; unattended: `awaiting_user` + в список):
   - сообщение от человека, или сомнение бот/человек (лишний вопрос юзеру дешевле неверного авто-ответа живому рекрутеру); к согласованию готовить 2–3 драфта, разных по сути, не косметически;
   - ЛЮБОЙ вопрос по стеку/технологиям — кнопки И текст (потребовано юзером@2026-07-13: снять вопрос+варианты read-only → показать предлагаемый ответ по фактам → юзер подтверждает или пополняет candidate-facts → только потом отправка); кнопочные списки не про стек — тоже до клика;
   - требует ресурса юзера: созвон/выбор времени (даже от бота), объёмное тестовое («сделайте проект», репозиторий).
3. **АВТО** — по `profile/candidate-facts.md` + полный текст вакансии (`data/details/{id}.json`), стиль из profile.md; факт вне candidate-facts → честное «нет» или в рунг 2:
   - свободный текстовый вопрос бота НЕ про стек;
   - анкеты/формы/короткие эссе — Playwright-субагент. Барьеры: обязательное поле вне фактов без честного «нет» → СТОП анкеты, вопрос дословно в отчёт; ОПЦИОНАЛЬНОЕ поле вне фактов → пропустить поле (не всю анкету), отметить в отчёте; требует аккаунт/пароль/платёжные РЕКВИЗИТЫ (вопрос о зарплатных ожиданиях — не барьер, ответ из candidate-facts) → СТОП, юзеру ссылку; форма во вложенном iframe → открывать top-level по src (тот же OOPIF-урок, случай@2026-07-19 Tilda); submit только после скриншота заполненного.

## step-3: Отправка (opus-субагент, headed Playwright)

Отправка в чат — ТОЛЬКО браузером (chatik REST на запись недоступен). Сессия и общие правила браузера — hh-test-apply step-3 (куки browser_cookie3 → add_cookies; предупредить юзера не кликать в окне; CAPTCHA → стоп). Каждую отправку верифицировать появлением нового уникального `chatik-chat-message-<id>`, скриншот до/после.

Выстрадано (передавать субагенту в промпт):
- Перед ЛЮБОЙ отправкой перечитать чат живьём: (а) диалог закрыт ботом (маркеры рунга МОЛЧАНИЕ) → не отправлять — поле ввода в закрытом чате технически доступно, это ловушка (потребовано юзером@2026-07-19); (б) в чате ЛЮБОЕ сообщение/клик, которого нет в снимке (юзер отвечает и с телефона — случилось@2026-07-19: consent-кнопку нажал сам) → не отправлять по старому плану, чат вернуть на переклассификацию.
- Чат открывать top-level: `https://chatik.hh.ru/chat/<chatId>` (`chatUrl` из inbox.json; редирект на `<регион>.hh.ru/chat/…` — норма). Iframe на negotiations-странице — cross-origin OOPIF, `page.frames` его не берёт (выстрадано@2026-07-10).
- Баннер `[data-qa="cookies-policy-informer"]` перекрывает поле ввода и перехватывает клики — скрыть/принять до ввода.
- Селекторы: ввод `textarea[data-qa="chatik-new-message-text"]`, отправка `button[data-qa="chatik-do-send-message"]`, сообщение `[data-qa="chatik-chat-message-<id>"]`. Префикс-селектор двоится — контейнер + вложенный `…-text`; дубль-контроль только по уникальным id (выстрадано@2026-07-13).
- Бот «Робот-рекрутер» свободный текст НЕ парсит: в анкету работодателя попадает только клик по кнопке, текст остаётся виден работодателю → связка: развёрнутый честный текст + клик правдивой кнопки. Кнопки бота без data-qa (magritte-button) — матчить exact-текстом; suggestion-чипы hh («Какой график работы?» и т.п.) — шаблоны НАШИХ сообщений, не вопросы бота, не кликать. networkidle в чатах не наступает (websocket) — ждать фиксированно.

## step-4: Лог

`data/applied.json` — единственная state-машина переписки:
- `status` — закрытый enum: `sent → replied → awaiting_user → interview | rejected | closed`. Процессные события (скрининг завершён, анкета сдана, чат покинут) — не статусы, а `kind: event` в dialog. Новое значение enum — только правкой этого файла.
- `lastContact` (ISO) — курсор needsAttention; обновлять при КАЖДОЙ обработке чата.
- `dialog` — append строго в схеме `{at: ISO, from: user|employer|bot|system, kind: q|a|note|event, text}`, других ключей не заводить.
- Сводка юзеру каждый прогон обязана содержать секцию «Ждёт юзера»: фильтр `status: awaiting_user` по ВСЕМУ applied.json (не только сегодняшнее), позиция = с какой даты + что нужно + ссылка.

## Ошибки

- inbox.py пуст/упал → куки протухли (залогиниться в Chrome на hh.ru) или chatik/negotiations сменили формат — проверить руками GET из `inbox.py::CONFIG_URL` и `hh.ru/applicant/negotiations?filter=all`.
