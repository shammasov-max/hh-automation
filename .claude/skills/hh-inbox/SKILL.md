# hh-inbox: обработка входящих по откликам

Рабочий каталог: `/home/max/automations/hh`.

## step-1: Сбор

```bash
python3 src/inbox.py
```

Сводка stdout — рабочий вход, `data/inbox.json` целиком НЕ читать:
- `attention[]` — чаты к обработке; поля: vacancyId/name/company/chatId/chatUrl/state/status/needsAttention/stateMismatch/unread/error/`lastMsg` (последнее сообщение: id/at/mine/bot/name/state/text; `lastMsg.id` — готовый `--expect-last-id`). Полная история — в inbox.json у той же записи (поле `history`), читать точечно по vacancyId. `error` заполнен → чат не прочитался, разобрать руками.
- `autoRejected[]` — DISCARD, скрипт уже перевёл в rejected (имена — в inbox.json).
- `awaitingUser[]` — заготовка секции «Ждёт юзера» (id/name/company/since; chatUrl и вопрос — в applied.json).
- `…[обрезано]` в тексте history → сообщение длиннее 1500; дочитать:
  ```bash
  python3 -c "import sys,json; sys.path.insert(0,'src'); from hh_session import make_session; from chatik import chat_data; print(json.dumps(chat_data(make_session(), CHATID)['chat']['messages']['items'], ensure_ascii=False))"
  ```

## step-2: Классификация (главный тред; сверху вниз, first match)

Перед лестницей: `stateMismatch` c `state: INTERVIEW` = приглашение — ВСЕГДА рунг 2 (даже без вопроса в тексте) + отдельная секция «Приглашения» в сводке; `status: interview` ставить только после решения юзера.

1. **МОЛЧАНИЕ** — в чат не писать и не кликать НИЧЕГО; лог = `applied_log --id X` (сдвиг курсора; статус — по таблице step-4):
   - диалог закрыт ботом: «ответы отправлены работодателю», «ИИ-помощник завершил работу», итоговая сводка ответов, «пользователь … покинул чат» — даже если выше висят неотвеченные вопросы (потребовано юзером@2026-07-19: ответ никуда не попадёт);
   - отказ: «не готовы пригласить» и т.п. → `--status rejected`; никаких поправок/уточнений задним числом (потребовано юзером@2026-07-13). Исключение: отказ со ВСТРЕЧНЫМ вопросом работодателя — это не поправка, а новый вопрос → рунг 2;
   - инфо без вопроса («передали рекрутеру», «рассмотрим») → текст в dialog заметкой (`employer/note`) — это же след для будущей чистки лимбо.
2. **СОГЛАСОВАНИЕ** — interactive (дефолт): AskUserQuestion; unattended (`/hh-inbox auto` или юзер велел не спрашивать): `--status awaiting_user`, вопрос → dialog (`bot|employer`/`q`), драфты → dialog (`user`/`note`), позиция в «Ждёт юзера»:
   - сообщение от человека (`lastMsg.bot: false`) или сомнение бот/человек — лишний вопрос юзеру дешевле неверного авто-ответа живому рекрутеру; к согласованию 2–3 драфта, разных по сути;
   - ЛЮБОЙ вопрос по стеку/технологиям — кнопки И текст (потребовано юзером@2026-07-13: снять вопрос+варианты read-only → показать предлагаемый ответ по фактам → юзер подтверждает или пополняет candidate-facts → только потом отправка); любые кнопки в ЧАТЕ — тоже до клика;
   - требует ресурса юзера: созвон/выбор времени (даже от бота), объёмное тестовое.
3. **АВТО** — ТОЛЬКО при `bot: true` у последнего чужого сообщения (bot: false → рунг 2). Ответ по `profile/candidate-facts.md` + полный текст вакансии (`data/details/{id}.json`; нет файла → `python3 src/fetch_vacancy.py --ids <id>`; не скачалась — по history+фактам, пометить в отчёте). Факт вне candidate-facts → честное «нет» или рунг 2:
   - свободный текстовый вопрос бота НЕ про стек;
   - анкеты/формы по ВНЕШНЕЙ ссылке (не кнопки в чате) — Playwright-субагент по шаблону step-3 (стоп-барьеры зашиты в шаблон).

## step-3: Отправка

**Текст в чат — REST.** Dry-run ОБЯЗАТЕЛЕН перед каждым send:

```bash
python3 src/chat_send.py --chat-id <chatId> --text-file <f> --expect-last-id <lastMsg.id>          # dry-run
python3 src/chat_send.py --chat-id <chatId> --text-file <f> --expect-last-id <lastMsg.id> --send
```

- `--expect-last-id` обязателен (пустая история → `--expect-empty`); скрипт сам откажется, если чат ушёл вперёд (юзер отвечает и с телефона — случилось@2026-07-19), закрыт hh или запись в статусе rejected/closed (`--override-status` — только для согласованного «отказа со встречным вопросом»). Маркеры «бот закрыл диалог» скрипт не знает — их LLM проверяет по history ДО вызова.
- `result: sent` при ЛЮБОМ `verified` (true/false/unknown) = ушло, НЕ ПОВТОРЯТЬ вызов. `newAfterSend` в выводе — чужие сообщения после нашего (бот отвечает в ту же секунду): обработать как attention сейчас же, не терять.
- Отправка сама логирует dialog + lastContact (= время своего сообщения); статус — отдельно `applied_log --status`.
- В сводке юзеру у каждого АВТО-ответа — строка-источник из candidate-facts (галлюцинированный факт хуже молчания).

**Кнопки бота и анкеты — Playwright-субагент** (opus, headed). Промпт собирать из шаблона (не пересказывать):

## step-4: Лог

`data/applied.json` — state-машина; писать ТОЛЬКО через `src/applied_log.py`, сразу после обработки КАЖДОГО чата (не батчем). `lastContact` = `at` ПОСЛЕДНЕГО обработанного сообщения, не «now» (chat_send при отправке ставит сам).

Статус: наш ответ отправлен → `replied`; ждёт юзера → `awaiting_user`; приглашение подтверждено юзером → `interview`; отказ → `rejected`; бот закрыл диалог / чат покинут → `closed` (сам факт — event). Dialog (одна запись = один вызов CLI): вопрос бота → `bot/q`; вопрос человека → `employer/q`; наш ответ → `user/a`; инфо/заметка → `employer/note`, наша — `user/note`; служебное → `system/event`.

Сводка юзеру каждый прогон: отказы (вкл. `autoRejected` с именами), авто-отвечено (+источник факта), «Приглашения» (INTERVIEW), согласовано-отправлено, «Ждёт юзера» = `awaitingUser` + что нужно + chatUrl.

## Чистка чатов (ТОЛЬКО по команде юзера: «почисти чаты»)

Выход из чата: negotiation цела, чат остаётся writable (обратимо), но работодатель видит событие выхода — потому не в авто-прогоне. Критерии (прецедент@2026-07-15, 106 чатов):
- отказы — батчем: `python3 src/chat_leave.py --rejected` (dry-run) → показать список → `--send`;
- лимбо «обдумаем/вернёмся» — источник только dialog-заметки (`note`); чего нет в dialog — не лимбо, не выдумывать; список на подтверждение юзеру → `--ids …`;
- НЕ выходить: interview, awaiting_user (скрипт блокирует всегда), sent/replied без `--force`, чаты с неотвеченным вопросом к нам (проверка LLM по history до `--force`).
