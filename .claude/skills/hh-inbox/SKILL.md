# hh-inbox: обработка входящих по откликам

Рабочий каталог: `/home/max/automations/hh`.

## step-1: Сбор

```bash
python3 src/inbox.py
```

Сводка stdout — рабочий вход, `data/inbox.json` целиком НЕ читать:
- `attention[]` — чаты к обработке (needsAttention: новое чужое сообщение; stateMismatch: hh прислал INTERVIEW). Полная история чата — в inbox.json у этой записи (поле `history`), читать точечно по vacancyId.
- `autoRejected[]` — DISCARD, скрипт уже перевёл в rejected (для сводки юзеру).
- `awaitingUser[]` — заготовка секции «Ждёт юзера».
- `…[обрезано]` в тексте history → сообщение длиннее 1500, дочитать: `python3 -c` с `chatik.chat_data`.

## step-2: Классификация (главный тред; сверху вниз, first match)

1. **МОЛЧАНИЕ** — только лог, в чат не писать и не кликать НИЧЕГО:
   - диалог закрыт ботом: «ответы отправлены работодателю», «ИИ-помощник завершил работу», итоговая сводка ответов, «пользователь … покинул чат» — даже если выше висят неотвеченные вопросы (потребовано юзером@2026-07-19: ответ никуда не попадёт);
   - отказ: «не готовы пригласить» и т.п. → `status: rejected`; никаких поправок/уточнений задним числом (потребовано юзером@2026-07-13). Исключение: отказ со ВСТРЕЧНЫМ вопросом работодателя — это не поправка, а новый вопрос → рунг 2;
   - инфо без вопроса («передали рекрутеру», «рассмотрим»).
2. **СОГЛАСОВАНИЕ** — interactive (дефолт): AskUserQuestion; unattended (`/hh-inbox auto` или юзер велел не спрашивать): `status: awaiting_user` + в секцию «Ждёт юзера»:
   - сообщение от человека, или сомнение бот/человек (лишний вопрос юзеру дешевле неверного авто-ответа живому рекрутеру); к согласованию 2–3 драфта, разных по сути, не косметически;
   - ЛЮБОЙ вопрос по стеку/технологиям — кнопки И текст (потребовано юзером@2026-07-13: снять вопрос+варианты read-only → показать предлагаемый ответ по фактам → юзер подтверждает или пополняет candidate-facts → только потом отправка); любые кнопки в ЧАТЕ — тоже до клика;
   - требует ресурса юзера: созвон/выбор времени (даже от бота), объёмное тестовое («сделайте проект», репозиторий).
3. **АВТО** — по `profile/candidate-facts.md` + полный текст вакансии (`data/details/{id}.json`), стиль из profile.md; факт вне candidate-facts → честное «нет» или в рунг 2:
   - свободный текстовый вопрос бота НЕ про стек;
   - анкеты/формы по ВНЕШНЕЙ ссылке (не кнопки в чате) — Playwright-субагент. Барьеры: обязательное поле вне фактов без честного «нет» → СТОП анкеты, вопрос дословно в отчёт; ОПЦИОНАЛЬНОЕ поле вне фактов → пропустить поле (не всю анкету), отметить в отчёте; требует аккаунт/пароль/платёжные РЕКВИЗИТЫ (вопрос о зарплатных ожиданиях — не барьер, ответ из candidate-facts) → СТОП, юзеру ссылку; форма во вложенном iframe → открывать top-level по src (OOPIF-урок, случай@2026-07-19 Tilda); submit только после скриншота заполненного.

## step-3: Отправка

**Текст в чат — REST, браузер не нужен:**

```bash
python3 src/chat_send.py --chat-id <chatId> --text-file <f> --expect-last-id <id>          # dry-run
python3 src/chat_send.py --chat-id <chatId> --text-file <f> --expect-last-id <id> --send
```

`--expect-last-id` = id последнего сообщения history при классификации, давать ВСЕГДА — скрипт сам откажется, если чат ушёл вперёд (юзер отвечает и с телефона — случилось@2026-07-19) или закрыт со стороны hh. Маркеры «бот закрыл диалог» (рунг МОЛЧАНИЕ) скрипт не знает — их LLM проверяет по history ДО вызова.

**Кнопки бота и анкеты hh — Playwright-субагент** (opus, headed; сессия и общие правила браузера — hh-test-apply step-3, путь `.claude/skills/hh-test-apply/SKILL.md`). Выстрадано, передавать субагенту в промпт:
- Перед действием перечитать чат живьём; любое сообщение/клик, которого нет в снимке → стоп, чат на переклассификацию.
- Чат открывать top-level: `https://chatik.hh.ru/chat/<chatId>` (`chatUrl` из inbox.json; редирект на `<регион>.hh.ru/chat/…` — норма). Iframe на negotiations-странице — cross-origin OOPIF, `page.frames` его не берёт (выстрадано@2026-07-10).
- Баннер `[data-qa="cookies-policy-informer"]` перекрывает поле ввода и перехватывает клики — скрыть/принять до ввода.
- Селекторы: ввод `textarea[data-qa="chatik-new-message-text"]`, отправка `button[data-qa="chatik-do-send-message"]`, сообщение `[data-qa="chatik-chat-message-<id>"]`. Префикс-селектор двоится — контейнер + вложенный `…-text`; дубль-контроль только по уникальным id (выстрадано@2026-07-13).
- Бот «Робот-рекрутер» свободный текст НЕ парсит: в анкету работодателя попадает только клик по кнопке, текст остаётся виден работодателю → связка: развёрнутый честный текст + клик правдивой кнопки. Кнопки бота без data-qa (magritte-button) — матчить exact-текстом; suggestion-чипы hh («Какой график работы?» и т.п.) — шаблоны НАШИХ сообщений, не вопросы бота, не кликать. networkidle в чатах не наступает (websocket) — ждать фиксированно.

## step-4: Лог

`data/applied.json` — state-машина переписки; писать ТОЛЬКО через `src/applied_log.py` (схема dialog, enum статусов и валидация lastContact зашиты в него), сразу после обработки КАЖДОГО чата, не батчем в конце.

Статус по результату обработки: наш ответ отправлен → `replied`; ждёт решения юзера → `awaiting_user`; приглашение принято в работу → `interview`; текстовый отказ без DISCARD → `rejected`; бот закрыл диалог / чат покинут → `closed` (сам факт — `kind: event` в dialog).

Сводка юзеру каждый прогон: отказы (вкл. `autoRejected`), авто-отвечено, согласовано-отправлено, секция «Ждёт юзера» = `awaitingUser` из сводки + что нужно + ссылка.

## Ошибки

- inbox.py пуст/упал → куки протухли (залогиниться в Chrome на hh.ru) или hh сменил формат — проверить руками `hh.ru/applicant/negotiations?filter=all` и `chatik.chat_data`.
- `chat_send.py` → `refused: race` — в чате новое сообщение, вернуть чат на step-2.
