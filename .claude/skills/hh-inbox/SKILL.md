# hh-inbox: обработка входящих по откликам

Рабочий каталог: `/home/max/automations/hh`.

## step-1: Сбор

```bash
python3 src/inbox.py
```

→ `data/inbox.json`: чаты по нашим откликам (`applied.json`), с последним сообщением, `unread`, `state` (DISCARD/INTERVIEW/…), `isBot`. Полной истории чата этот канал не даёт — только последнее сообщение; для длинного контекста чат откроет Playwright-субагент (step-4).

Обрабатывать: `unread > 0` ИЛИ `state` изменился с прошлого прогона, И `lastMessage.isMine == false`.

## step-2: Классификация каждого сообщения (главный тред, Opus)

| Категория | Признаки | Действие |
|-----------|----------|----------|
| **Отказ** | `state: DISCARD`, «не готовы пригласить», «выбрали других» | лог в applied.json (`status: rejected`), НЕ отвечать; переписку НЕ продолжать вообще — никаких поправок/уточнений задним числом (потребовано юзером@2026-07-13) |
| **ИИ/бот-вопрос** | `isBot: true`, «ИИ-помощник», длинный структурированный опросник, анкетные вопросы списком, шаблон без персонализации | свободный текст НЕ про стек — **авто-ответ** по `profile/candidate-facts.md`; **вопросы по стеку/технологиям (любые: кнопки И текст) — ВСЕГДА сначала к юзеру** (потребовано юзером@2026-07-13: снять вопрос+варианты read-only → показать юзеру предлагаемый ответ по фактам → юзер подтверждает или пополняет candidate-facts → только потом отправка). Кнопочные списки без стека — тоже согласовать до клика |
| **Человеческий вопрос** | короткий (< ~300 симв.), разговорный, конкретный, от живого имени, персональный контекст | **2–3 драфта → AskUserQuestion → отправить выбранный.** Без согласования НЕ отправлять |
| **Приглашение на созвон/интервью** | выбор времени, календарь, «когда удобно» | ВСЕГДА согласование (время юзера), даже если от бота |
| **Анкета / чек-лист / опросник** | ссылка на форму, «заполните анкету», список вопросов в чате | **авто-заполнение** по candidate-facts (Playwright-субагент, паттерн hh-test-apply). Стоп-барьеры: вопрос без покрытия фактами и без честного «нет» → пропустить и в отчёт; форма требует создать аккаунт/пароль/платёжные данные → СТОП, юзеру ссылку |
| **Объёмное тестовое задание** | «сделайте проект», репозиторий, многочасовая разработка | согласование (трудозатраты юзера); короткие эссе/архитектурные ответы в рамках формы — авто |
| **Инфо без вопроса** | «передали рекрутеру», «рассмотрим» | лог, не отвечать |

Сомнение между ИИ и человеком → трактовать как человеческий (согласование). Внешняя отправка дешевле лишнего вопроса юзеру, чем неверный авто-ответ живому рекрутеру.

## step-3: Подготовка ответов

- Авто-ответы (ИИ-категория): по фактам `candidate-facts.md` + полный текст вакансии (`data/details/{id}.json`); стиль писем из `profile.md` — конкретно, без воды, от первого лица. Факт вне candidate-facts → честное «нет» или перевести в категорию согласования.
- Драфты для человеческих вопросов: 2–3 варианта тоном под сообщение (короткий вопрос — короткий ответ), различающиеся по сути (например: «назвать вилку» vs «спросить детали до цифры»), не косметически.

## step-4: Отправка (opus-субагент, Playwright)

Отправка сообщений в чат — ТОЛЬКО браузером (chatik REST для записи недоступен). Agent (**model: opus**, headed, паттерн из `hh-test-apply` step-3: куки browser_cookie3 → add_cookies): для каждого чата открыть `chatUrl` из inbox.json, при необходимости дочитать историю, вставить согласованный/авто текст в поле сообщения, отправить, скриншот до/после, верифицировать появление сообщения в чате. Предупредить юзера не кликать в окне. CAPTCHA → стоп.

Выстрадано@2026-07-10 (передавать субагенту в промпт):
- Чат открывать top-level: `https://chatik.hh.ru/chat/<chatId>` (редирект на `<регион>.hh.ru/chat/…`). Iframe чата на странице negotiations — cross-origin OOPIF, через `page.frames` не берётся.
- Баннер `[data-qa="cookies-policy-informer"]` перекрывает поле ввода и перехватывает клики — скрыть/принять до ввода.
- Селекторы: ввод `textarea[data-qa="chatik-new-message-text"]`, отправка `button[data-qa="chatik-do-send-message"]`, сообщения `[data-qa="chatik-chat-message-<id>"]`.
- Селектор сообщений двоится (выстрадано@2026-07-13): `[data-qa^="chatik-chat-message"]` даёт 2 элемента на одно сообщение (контейнер + вложенный `…-text`) — проверку «не задублилась ли отправка» вести по уникальным id `chatik-chat-message-<id>`, не по сырым совпадениям.
- Бот hh «Робот-рекрутер» свободный текст НЕ парсит: повторяет вопрос с quick-reply-кнопками, в анкету работодателя попадает только клик по кнопке. Текст при этом остаётся виден работодателю → рабочая связка: развёрнутый текст (для человека) + клик по правдивой кнопке (для бота). Кнопки бота без data-qa (magritte-button) — матчить exact-текстом; suggestion-чипы hh («Какой график работы?» и т.п.) выглядят так же — это шаблоны НАШИХ сообщений, не вопросы бота, не кликать. networkidle в чатах не наступает (websocket) — ждать фиксированно.

## step-5: Лог

- `data/applied.json`: обновить записи — `status` (rejected/interview/replied), `lastContact` (ISO), `dialog` (append: вопрос → ответ).
- Юзеру сводка: отказы, авто-отвечено (что именно), согласовано-отправлено, ждёт его действий (анкеты/созвоны со ссылками).

## Ошибки

- inbox.py пуст/упал → куки протухли (залогиниться в Chrome) или chatik сменил формат конфига (`chats.chats.items`) — проверить руками GET из `src/inbox.py::CONFIG_URL`.
- Сообщение длиннее 2000 симв. обрезано в inbox.json → полный текст читает Playwright-субагент из чата.
